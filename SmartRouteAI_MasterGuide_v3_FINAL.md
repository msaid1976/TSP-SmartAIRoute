## How this workflow operates

Think of it like a construction project:

- **Spec Kit** = the architect's blueprint — defines exactly what gets built in each phase
- **Codex** = the construction crew — reads the blueprint and writes all the code
- **Claude Code** = the building inspector — checks every phase before you approve it
- **You** = the project owner — you confirm each phase gate and keep things moving
- **Git** = your safety net — every confirmed phase is saved as a permanent checkpoint

The process for every phase, without exception:

```
1.  Tell Codex which phase to implement
2.  Codex writes the code (5–15 minutes)
3.  You run and test it manually
4.  You paste results into Claude Code for review
5.  Claude approves or flags issues
6.  You give issues back to Codex to fix
7.  All clear → commit to Git → move to next phase
```

In VS Code, create a new file at the project root called `AGENTS.md`. Paste this exactly:

```markdown
# SmartRoute AI — Codex Agent Instructions

## Read before starting any work
1. Read .agents/skills/specification.md completely before writing any code.
2. For GraphCanvas work, read specs/references/SmartRouteAI_GraphCanvas_Design.md.
3. For Phase 006B (OCR), read specs/references/SmartRouteAI_OCR_ImageExtraction_Design.md fully.
4. For algorithm implementations, read the notebooks in specs/references/notebooks/ for IDEAS ONLY.
   Do NOT copy the VeRoLog domain classes: Gene, VeRoLogProblem, VeRoLogSolution,
   ChromosomeDecoder, BestFitDecreasingPacking, VeRoLogDataParser. These are specific to
   logistics scheduling and have no place in a generic routing platform.

## Project identity
SmartRoute AI is a GENERIC routing optimization SaaS platform.
It accepts any routing problem expressible as: nodes + distances + objectives.
It does NOT model machines, technicians, trucks, delivery dates, or installation dates.
Users: anyone with a routing problem — cities, warehouses, delivery stops, abstract graphs.

## Technology stack (frozen — do not change without explicit approval)
- Frontend: Next.js 14 App Router, TypeScript strict mode, Tailwind CSS, shadcn/ui, Recharts, react-konva
- Backend: FastAPI, Pydantic v2, PostgreSQL 15, Redis 7, Celery 5
- Solvers: OR-Tools, GA, ACO, SA, PSO, NSGA-II
- Infrastructure: Docker Compose, Alembic migrations, pnpm workspaces

## Generic problem model (never add domain-specific fields)
Every problem normalizes to exactly:
  problemType: "tsp | atsp | weighted_tsp | matrix_tsp | open_tsp"
  nodes: [{id, label, x, y}]          ← x/y can be null for matrix_tsp
  distanceMatrix: [[...]] | null       ← null when coords are available
  startNodeId: string
  returnToStart: boolean
  objective: "min_distance | min_cost | min_time"
  constraints: {}

No machine_id. No technician_id. No delivery_date. No installation_date. No truck_capacity.

## BaseSolver contract (frozen — never change)
Every solver extends BaseSolver and implements:
  solve(problem: dict) → dict
  validate_route(route, nodes) → bool
  get_distance_matrix(problem) → np.ndarray
    # Uses distanceMatrix if present, otherwise builds from x/y coordinates
    # This is how all solvers handle both tsp AND matrix_tsp with zero extra code

## Algorithm ideas to use from notebooks
- GA notebook: adaptive crossover switching by generation (OX → PMX), 2-opt post-processing, elitism
- PSO notebook: continuous float vector → rank decoding → valid permutation
- NSGA-II notebook: pymoo integration, knee-point extraction from Pareto front
- ACO notebook: multi-phase pipeline (ACO → SA → VNS), elite ant reinforcement, stagnation detection

## Solver registry
SOLVER_REGISTRY = { "ortools", "ga", "aco", "sa", "pso", "nsga2" }
All solvers extend BaseSolver in apps/api/solvers/base.py.
All solvers must: accept seed, record convergence[], call validate_route() before returning.

## Recommendation scoring formula
score = 0.40*quality + 0.20*runtime + 0.15*stability + 0.15*scalability + 0.10*constraint_fit
Rule-based only — never call an LLM inside the recommendation engine.

## Rules — always follow without exception
1. Implement ONE phase at a time — never implement Phase N+1 without explicit human approval
2. Stop at every phase gate and list all acceptance criteria for the human to verify
3. All Python: Pydantic v2 models, typed signatures, no bare dicts
4. All TypeScript: strict mode, zero `any` types
5. Every solver extends BaseSolver — no standalone solver scripts ever
6. Solvers ALWAYS run in Celery workers — never block the FastAPI thread
7. All secrets in .env — never hardcode credentials
8. Every new module needs tests: solvers get seeded unit tests, API gets integration tests
9. Run `git add . && git commit` after every confirmed phase

## UI design direction
- Dark-first: deep navy (#0A0F1E) background, electric blue (#3B82F6) accents, white text
- Typography: DM Sans for UI text, JetBrains Mono for data/numbers/routes
- shadcn/ui as component base, customized with Tailwind utility classes
- Smooth animated transitions on all route maps and convergence charts
- Mobile-responsive from Phase 005 onward

  - Order Crossover (OX) for early generations, Partially Mapped (PMX) after gen 50 (adaptive)
  - Gaussian mutation: swap probability decays from 0.15 to 0.05 over generations
  - 2-opt local search applied to best individual each generation
  - Tournament selection (k=3), top 10% elitism (unchanged in next generation)
  - Default: population_size=100, generations=300, crossover_rate=0.8, mutation_rate=0.1
  - Record best distance in convergence[] every 10 generations

apps/api/solvers/aco.py — AntColonySolver (from ACO notebook ideas):
  - Pheromone matrix with minimum floor value (0.001) to prevent zeroing
  - Elite ant reinforcement: top 3 ants deposit 3× normal pheromone
  - Stagnation detection: if best cost unchanged for 20 iterations, increase rho by 0.05
  - 2-opt improvement pass on best solution after main ACO loop completes
  - Default: n_ants=50, n_iterations=100, alpha=1.0, beta=2.0, rho=0.1, Q=100
  - Record best distance in convergence[] every iteration

apps/api/solvers/ortools_solver.py — ORToolsSolver:
  - Uses ortools.constraint_solver routing model
  - CHRISTOFIDES first-solution strategy
  - supports_asymmetric = True (uses get_distance_matrix — handles matrix_tsp)
  - Timeout controlled by problem["constraints"].get("maxRuntimeSeconds", 60)
  - Record final distance in convergence[] as single entry

apps/api/solvers/__init__.py — SOLVER_REGISTRY dict

## PHASE 004 — All Solvers + Recommendation Engine

**What gets built:** SA, PSO, NSGA-II, Tabu Search, and Differential Evolution solvers, multi-run benchmarking, comparison engine with explanation, four summary cards.

### Codex — build it

```
Phase 003 confirmed. Implement Phase 004 — All Solvers and Recommendation Engine.

Before writing PSO, read specs/references/notebooks/GA-Heterogeneous_with_PSO.ipynb for PSO ideas.
Before writing NSGA-II, read specs/references/notebooks/GA_Heterogeneous_Design_with_NSGA_Pareto_Front.ipynb.

Extract algorithm IDEAS only. No VeRoLog domain classes.

New solvers (all extend BaseSolver, all use get_distance_matrix()):

apps/api/solvers/sa.py — SimulatedAnnealingSolver:
  - Start from nearest-neighbour greedy tour as initial solution
  - 2-opt neighbourhood: try swapping two edges, accept worse solution with probability exp(-Δ/T)
  - Geometric cooling: T = T0 × cooling_rate^iteration, stop at T_min = 0.01
  - Record current best distance in convergence[] every 100 iterations
  - Default: T0=1000, cooling_rate=0.995, max_iter=10000

apps/api/solvers/pso.py — ParticleSwarmSolver (from PSO notebook ideas):
  - Encode tour as continuous vector: each element is the node's visit rank (float)
  - Velocity update: v = w*v + c1*r1*(pbest-x) + c2*r2*(gbest-x)
  - Inertia weight w decays from 0.9 to 0.4 linearly over iterations
  - After each position update: decode float vector to valid permutation via argsort
  - Apply 2-opt to personal best and global best each iteration
  - Record best tour distance in convergence[] every iteration
  - Default: n_particles=50, n_iterations=200, c1=2.0, c2=2.0, w_max=0.9, w_min=0.4

apps/api/solvers/nsga2.py — NSGAIISolver (from NSGA-II notebook ideas):
  - Two objectives: minimize total_distance, minimize max_single_edge (route balance)
  - Use pymoo library: NSGA2 algorithm, FloatRandomSampling, SBX crossover, PM mutation
  - Encode as float permutation vector, decode to tour for evaluation
  - Extract knee-point solution from Pareto front as the single recommended route
  - Store full Pareto front [{route, distance, max_edge}] in result["solverParams"]["paretoFront"]
  - Use QuietLogger pattern to suppress all verbose output (this runs in a web context)
  - Default: pop_size=100, n_gen=200
  - supports_multi_objective = True

apps/api/solvers/tabu.py — TabuSearchSolver:
  - Maintain a tabu list as a circular buffer of the last tabu_tenure moves
  - Neighbourhood: all 2-opt edge-swap pairs (~N²/2 candidates per iteration)
  - Aspiration criterion: accept a tabu move if it produces a new global best
  - Intensification: every 50 iterations, restart from the best solution found so far
  - Diversification: if no improvement for diversification_threshold iterations,
    apply a 4-opt double-bridge perturbation to escape local optima
  - Record best distance in convergence[] every 10 iterations
  - Default: max_iter=2000, tabu_tenure=15, diversification_threshold=100

apps/api/solvers/de.py — DifferentialEvolutionSolver:
  - Encode each candidate tour as a continuous random-key vector (one float per node)
  - DE mutation strategy DE/rand/1/bin:
      mutant = x_r1 + F × (x_r2 − x_r3) where r1, r2, r3 are distinct random individuals
  - Binomial crossover: accept each dimension from mutant with probability CR
  - Greedy selection: offspring replaces parent only if it decodes to a shorter tour
  - Decode float vector → valid permutation via argsort at every evaluation
  - Apply 2-opt improvement to the best individual in each generation
  - Record best distance in convergence[] every 10 generations
  - Default: population=80, max_iter=500, F=0.8, CR=0.9

Update SOLVER_REGISTRY with all 8 solvers:
  {"ortools", "ga", "aco", "sa", "pso", "nsga2", "tabu", "de"}

apps/api/services/benchmarker.py:
  For N runs per solver, compute:
    avgDistance, bestDistance, worstDistance, stdDev
    avgRuntimeMs, bestRuntimeMs
    stabilityScore = 1.0 - (stdDev / avgDistance) clamped to [0.0, 1.0]
    feasibilityRate = (successful_runs / total_runs)
  Also compute summaryCards:
    best_distance_solver = solver with lowest bestDistance
    fastest_solver = solver with lowest avgRuntimeMs
    most_stable_solver = solver with highest stabilityScore
    recommended_solver = solver with highest final_score (from recommender)

apps/api/services/recommender.py (rule-based, zero LLM calls):
  Scoring formula:
    quality_score     = (best_distance - solver_distance) / (best_distance - worst_distance)
    runtime_score     = (worst_runtime - solver_runtime) / (worst_runtime - best_runtime)
    stability_score   = stabilityScore (from benchmarker)
    scalability_score = {"ortools":0.3,"ga":0.7,"aco":0.8,"sa":0.6,"pso":0.6,"nsga2":0.5,"tabu":0.7,"de":0.65}
    constraint_fit    = 1.0 if no constraints hit, else 0.7
    final_score = 0.40*quality + 0.20*runtime + 0.15*stability + 0.15*scalability + 0.10*constraint_fit

  Explanation template for recommended solver:
    "{solver} achieved the best overall score ({score:.2f}/1.00).
     Average route distance: {avg_distance:.1f} units in {avg_runtime_ms:.0f}ms
     across {n_runs} run(s). {stability_note}"
  Where stability_note:
    "Variance was low — a reliable choice." if stability_score > 0.9
    "Some route variance observed across runs." otherwise

  Trade-off text for second-best solver (used by TradeOffPanel):
    "If {priority} matters more than overall balance, consider {second_solver}
     ({second_metric} vs winner's {winner_metric})."
  Where priority is inferred from whichever sub-score the second solver wins.

New API:

  GET  /api/comparisons/{problemId} — returns full ComparisonResult including summaryCards

Alembic migration for comparisons table.

  apps/web/components/ParetoFrontChart.tsx — Recharts scatter plot of NSGA-II Pareto front

  - Return preprocessed image array

apps/api/services/ocr/classifier.py:
  - Classify image type: "map" | "table" | "sketch"
  - Use line detection (cv2.HoughLines) and blob count heuristics
  - "table" if many horizontal/vertical lines; "sketch" if many blobs; else "map"

apps/api/services/ocr/text_extractor.py:

def is_valid_route(route, problem):
    """Validity check for Hamiltonian cycle."""
    node_ids = {n["id"] for n in problem["nodes"]}

# PART 7 — TROUBLESHOOTING

| Problem | Symptom | Fix |
|---|---|---|
| WSL2 install fails | PowerShell shows error | Enable virtualization in BIOS (Intel VT-x or AMD-V). Restart after enabling. |
| Docker won't start | Whale icon spins forever | Restart PC. In Docker Desktop Settings, confirm WSL2 backend is selected. |
| Port 5432 already in use | `docker compose up` fails with "address in use" | Run `docker compose down` first, then `docker compose up -d` |
| `uvx` command not found | "uvx: command not found" | Run `source ~/.bashrc` then try again |
| `nvm` or `node` not found | "node: command not found" | Run `source ~/.bashrc`, then `nvm use 20` |
| pnpm install fails | Error about Node version | Run `nvm use 20` then retry pnpm install |
| Codex panel missing | Right sidebar is empty | Press `Ctrl+Shift+P`, type "Codex: Open", press Enter |
| Claude Code panel missing | No ⚡ icon in left sidebar | Press `Ctrl+Shift+P`, type "Claude Code: Open", press Enter |
| Alembic migration error | Alembic fails to apply | Run `docker compose down -v` (removes volumes), then `docker compose up -d` |
| Solver produces invalid route | `validate_route() returned False` | Check GA crossover for duplicates. Check ACO returns full permutation. |
| PSO route contains floats | Route is `[0.2, 1.7, 3.1, ...]` | The float-to-permutation decode step is missing — ask Codex to fix the argsort decode in pso.py |
| matrix_tsp fails on solver | `KeyError: 'x'` | `get_distance_matrix()` is not handling the distanceMatrix path — ask Codex to fix base.py |
| NSGA-II Pareto front empty | `paretoFront: []` | pymoo version mismatch — ask Codex to check requirements.txt and pin the pymoo version |
| Celery jobs stay "queued" | Jobs never start running | Run `docker compose logs worker` — look for Python import errors |
| TypeScript errors everywhere | Red underlines in all files | Run `cd apps/web && pnpm install` to restore dependencies |
| Codex writes in wrong folder | Files created in wrong location | Re-open VS Code with the SmartRouteAI root folder (not a subfolder) |
| Phase gate skipped | Codex keeps building Phase N+1 | Say: "Stop. You passed the phase gate. Do not continue. List the acceptance criteria for Phase N." |
| Git: undo a phase | Phase went wrong | `git log --oneline` to see history. `git revert HEAD` to undo the last commit. |
| OCR returns no candidates | Empty candidates array | Check Tesseract installed: `tesseract --version` in WSL terminal |
| VLM fallback fails silently | Tier 4 never triggers | Check http://localhost:5000/vlm/extract is running — Qwen3-VL-2B must be started separately |
